# E222 - merge_lane_clearance requires auto_lane_change

## Cause

A `theme` op set `merge_lane_clearance` while the resolved theme has
`auto_lane_change` disabled (its default). The clearance value only
governs how a migrating branch repacks around a merged or
pull-requested source's reserved lane — machinery that runs only under
`auto_lane_change`. With the flag off, no branch migrates, so the value
has no effect; accepting it silently would let the input read as if it
does something.

The trigger is *setting* the field, not setting it to a non-default
value — even `merge_lane_clearance: 1` (the default) under a disabled
flag is rejected, because the explicit-but-inert setting is itself the
mistake.

## Example

```
{"op": "theme", "merge_lane_clearance": 2}
```

```
{"op": "theme", "auto_lane_change": false, "merge_lane_clearance": 0}
```

## How to fix

Pick one of:

- Enable the feature the field configures: set `auto_lane_change: true`
  (then the clearance takes effect).
- Drop the `merge_lane_clearance` field, keeping the default behaviour.
- Switch to a named theme without `keep_prior_overrides: true` — the
  switch wipes prior overrides (including a now-irrelevant
  `merge_lane_clearance`), so the conflict clears on its own.
