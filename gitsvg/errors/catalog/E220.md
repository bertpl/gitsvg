# E220 - keep_prior_overrides requires a named theme

## Cause

A `theme` op set `keep_prior_overrides` (to `true` or `false`) without
also setting `name`. The flag only changes the behavior of a
named-theme switch — it controls whether previously-accumulated
`theme:` field overrides and `branch:` color overrides survive into
the new theme. On an op that doesn't set `name` there is no theme
switch happening, so the flag has no meaningful effect; accepting it
silently would let the input read as if it does something.

The flag's implicit default has no effect when `name` is absent —
only an *explicit* value triggers this error.

## Example

```
{"op": "theme", "keep_prior_overrides": true}
```

```
{"op": "theme", "keep_prior_overrides": false, "commit_radius": 8}
```

(In the second example, `commit_radius` still applies — the flag is
rejected, but other valid fields in the same op continue to
accumulate.)

## How to fix

Drop the `keep_prior_overrides` field, or add a `name` field so the
flag has something to modify. If the intent was "wipe my prior
overrides without changing theme," repeat the current theme's name
(e.g. `{"op": "theme", "name": "default"}`) — the default-false flag
will perform the wipe.
