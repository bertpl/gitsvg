# E207 - replaces: another branch is rooted on a replaced commit

## Cause

`replaces:` rule 4 — a `replaces:` squash atomically removes the
listed commits, but another branch was declared to fork from one of
them. After the squash, that branch would dangle.

## Example

```
{"op": "branch", "name": "main"}
{"op": "commit", "branch": "main", "id": "c1", "msg": "first"}
{"op": "branch", "name": "feat", "from_commit": "c1"}
{"op": "commit", "branch": "main", "id": "c2", "msg": "squash", "replaces": ["c1"]}
```

(`feat` is rooted on `c1`, but `c1` is being squashed)

## How to fix

`replaces:` is a strict shortcut for unambiguous cases. When other
branches depend on the commits you want to squash, use the
decomposed form: remove the dependent branches first (or rebuild
them after the squash), then remove the commits and add the new one
explicitly.
