# E208 - replaces: external commit's parents reference a replaced commit

## Cause

`replaces:` rule 5 — a commit outside the replaced set has a parent
referring to one of the commits you're trying to squash. After the
squash, that parent would dangle. Today this arises when a `merge`
commit's second parent — the tip of the branch it merged in — is later
squashed away on that branch.

## Example

```
{"op": "branch", "name": "main"}
{"op": "commit", "branch": "main", "id": "c1", "msg": "first"}
{"op": "branch", "name": "feat", "from_commit": "c1"}
{"op": "commit", "branch": "feat", "id": "f1", "msg": "work"}
{"op": "commit", "branch": "feat", "id": "f2", "msg": "more"}
{"op": "merge", "from": "feat", "into": "main", "as": "m1"}
{"op": "commit", "branch": "feat", "id": "f3", "msg": "squash", "replaces": ["f2"]}
```

(the merge `m1` has `f2` as its second parent; squashing `f2` would orphan it)

## How to fix

Don't squash a commit that another branch's `merge` still points at:
squash on the source branch *before* merging it, or rebuild the
dependent merge with the decomposed form (`remove` the merge, then a
fresh `merge`).
